# 8. 의사결정 기록

## 8.1 기술 선택 근거

### PySide6

데스크톱 환경에서 교사가 사용할 수 있는 GUI 애플리케이션을 만들기 위해 PySide6를 사용하였다.

선택 이유:

- Python 기반으로 개발 가능
- 데스크톱 UI 구성에 적합
- 테이블, 콤보박스, 다이얼로그, 버튼 등 업무용 UI 구현이 용이
- 로컬 DB와 연동하기 쉬움

### SQLite

로컬 데이터 저장을 위해 SQLite를 사용하였다.

선택 이유:

- 별도 서버 없이 사용 가능
- 설치와 배포가 간단함
- 학생, 문제, 시험, 결과 데이터를 저장하기에 충분함
- 교사용 단일 PC 환경에 적합함

### MVC 구조

UI와 Core 로직을 분리하기 위해 MVC 구조를 적용하였다.

선택 이유:

- UI 브랜치와 Core 브랜치 병합 시 역할 구분 가능
- View에서 DB 접근을 막을 수 있음
- Controller를 통해 기능 연결을 명확히 관리 가능
- Repository 변경이 View에 직접 영향을 주지 않음

### Repository 패턴

DB 접근 코드를 Repository에 모아 관리하였다.

선택 이유:

- SQL 로직을 한 곳에 집중
- Controller/View 코드 단순화
- 테이블별 CRUD 관리 용이

### Service 계층

시험지 생성, 채점, 분석처럼 복잡한 로직은 Service에서 처리한다.

선택 이유:

- Controller가 비대해지는 것을 방지
- 비즈니스 로직 재사용 가능
- 테스트 및 수정 범위가 명확해짐

## 8.2 대안 검토

### 웹 애플리케이션 방식

대안:

- React / Vue + 백엔드 서버 + DB

검토 결과:

- 여러 사용자가 동시에 쓰기에는 유리함
- 그러나 서버 구축과 배포가 필요함
- 현재 프로젝트 목적은 교사용 로컬 데스크톱 앱이므로 과함

결론:

- 채택하지 않음

### 파일 기반 저장 방식

대안:

- CSV / JSON 파일 저장

검토 결과:

- 구현은 단순함
- 하지만 학생, 문제, 시험, 결과 간 관계 관리가 어려움
- 검색, 필터링, 수정, 분석 기능이 복잡해짐

결론:

- 채택하지 않음

### View에서 직접 DB 호출

대안:

- View → SQLite

검토 결과:

- 빠르게 구현할 수 있음
- 하지만 MVC 구조가 깨지고 유지보수가 어려움
- UI 수정이 DB 로직에 영향을 줄 수 있음

결론:

- 채택하지 않음

### 수동 채점 방식

대안:

- 교사가 직접 정답/오답 입력

검토 결과:

- 구현은 단순함
- 채점 시간이 오래 걸림
- 실수 가능성이 높음

결론:

- 자동 채점 방식 채택

## 8.3 주요 변경 결정

### 1. View에서 Repository 직접 호출 금지

학생 관리, 문제은행, 시험지 생성, 결과 입력 기능 모두 View가 Repository를 직접 호출하지 않도록 수정하였다.

결정 이유:

- MVC 구조 유지
- 기능 흐름 명확화
- 유지보수성 향상

### 2. 학생 등록/수정/활성화 상태 변경을 Controller로 연결

학생 관리 UI 버튼이 Controller를 통해 Repository 기능을 호출하도록 변경하였다.

결정 이유:

- View와 DB 로직 분리
- 학생 목록 갱신 흐름 통일

### 3. 문제은행 샘플 데이터를 실제 DB 데이터로 교체

문제 목록을 QuestionController와 QuestionRepository를 통해 조회하도록 변경하였다.

결정 이유:

- 실제 저장 데이터 기반 화면 표시
- 검색/필터/상태 변경 기능 연동

### 4. 시험지 생성 화면을 Core 로직과 연결

자동 추출, 직접 선택, PDF 출력, 시험지 저장 기능을 Controller와 Service를 통해 연결하였다.

결정 이유:

- 문제 추출 로직 재사용
- View와 시험지 생성 Core 로직 분리
- 저장된 시험지 목록 관리 가능

### 5. 결과 입력 화면에서 답안 저장 및 자동채점 통합

답안 저장 버튼을 답안 저장 및 자동채점 흐름으로 변경하였다.

결정 이유:

- 사용자가 저장 후 별도 채점을 누르는 불편 제거
- 저장 누락으로 분석 결과가 빠지는 문제 방지

### 6. CSV 인코딩 fallback 적용

CSV 파일을 utf-8-sig, cp949, euc-kr 순서로 읽도록 변경하였다.

결정 이유:

- 엑셀에서 저장한 한글 CSV 호환성 확보

### 7. 결과 분석 요약 카드 key 기반 매핑 적용

카드 순서가 아니라 wrong_rate, correct_rate 같은 key를 기준으로 값을 표시하도록 수정하였다.

결정 이유:

- 전체 오답률 카드에 평균 정답 수가 표시되는 오류 방지

### 8. 대시보드 Core 로직 연결

DashboardController와 AnalysisService의 대시보드 분석 데이터를 DashboardView에 연결하였다.

결정 이유:

- 정적 샘플 화면을 실제 DB 기반 화면으로 전환
- 학생별 결과와 요약 지표 확인 가능

### 9. 대시보드 CSV 내보내기 한글 헤더 적용

CSV 헤더를 영어 key가 아니라 한글 컬럼명으로 출력하도록 변경하였다.

결정 이유:

- 교사가 바로 읽기 쉬운 결과 파일 제공

### 10. 화면 크기 차이 대응

주요 화면에 스크롤과 최소 너비, 콤보박스 팝업 스타일을 적용하였다.

결정 이유:

- 팀원 PC마다 UI가 잘리는 문제 완화
- 해상도/DPI 차이에 따른 표시 오류 감소
