# 6. 위험관리

## 6.1 주요 위험 요소

### 1. View에서 DB 또는 Repository를 직접 호출할 위험

MVC 구조가 깨지면 유지보수가 어려워지고, UI 변경 시 DB 로직까지 영향을 받을 수 있다.

### 2. UI 브랜치와 Core 브랜치 병합 충돌

View와 Controller가 동시에 수정되는 경우 충돌이 자주 발생할 수 있다.

### 3. 학생 반 이름 불일치

예를 들어 1학년1반과 1학년 1반처럼 공백 차이가 있으면 학생 조회나 결과 분석에서 누락이 발생할 수 있다.

### 4. 답안 저장과 채점 흐름 불일치

답안 입력 후 저장하지 않으면 분석 결과에 반영되지 않는 문제가 발생할 수 있다.

### 5. CSV 인코딩 문제

엑셀에서 저장한 CSV가 utf-8이 아니라 cp949 또는 euc-kr일 수 있어 파일 읽기 오류가 발생할 수 있다.

### 6. UI 크기 및 DPI 차이

팀원 PC마다 해상도, 배율, 폰트 렌더링이 달라 입력칸이나 콤보박스가 잘릴 수 있다.

### 7. 분석 카드 값 매핑 오류

요약 카드가 순서 기반으로 값을 표시하면 전체 오답률에 평균 정답 수가 표시되는 등 잘못된 결과가 보일 수 있다.

### 8. 생성 파일 커밋 위험

PDF나 CSV 같은 산출물이 Git에 포함될 수 있다.

## 6.2 대응 방안

- View는 화면 표시와 입력값 제공만 담당한다.
- DB 접근은 Repository에서만 수행한다.
- Controller는 View와 Core 로직 연결만 담당한다.
- 반 이름 비교 시 공백 제거 등 정규화 처리를 적용한다.
- 답안 입력 후 자동 저장 또는 저장 및 자동채점 흐름을 적용한다.
- CSV 읽기 시 utf-8-sig, cp949, euc-kr 인코딩을 순차적으로 지원한다.
- 주요 화면에 스크롤 영역과 최소 너비를 적용한다.
- 콤보박스 팝업 스타일과 최소 폭을 지정해 PC별 표시 차이를 줄인다.
- 분석 요약 카드는 순서 기반이 아니라 key 기반으로 표시한다.
- CSV/PDF 산출물은 커밋하지 않고 코드 파일만 관리한다.
